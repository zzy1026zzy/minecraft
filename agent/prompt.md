这个思路非常专业。在开启像 Trae 这样的 AI IDE 辅助开发时，**“先对齐宏观架构，再死磕微观实现”**是防止代码在后期重构中崩溃的唯一路径。特别是对于体素引擎，如果在打地基时没有考虑到未来多维度、高并发的系统交互，后期的拓展（比如加入复杂的火器弹道或者宠物寻路）将会带来灾难性的性能瓶颈。

我们把前置设定（阶段零）和前两个阶段（原版体验复现）进行深度拆解，并融入底层优化的考量。

阶段零：全局视野与架构定调 (向 AI 注入上下文)
在让 AI 写任何一行代码之前，必须先将整个项目的最终形态“喂”给它，让它在设计基类时预留好接口。

Prompt 0 - 项目初始化与架构约束：

“Trae，接下来的时间里，我们将使用 Unity (C#) 从零开发一款名为‘终极版’的体素生存游戏。

【项目宏观愿景】
它不仅需要完美复现原版 Minecraft 的核心体验（包括无限地形生成、方块破坏/放置、3x3合成、数据驱动背包），还要在未来无缝扩展以下高级系统：

多世界架构： 支持动态加载/卸载不同维度（如天堂、暮色森林），不同维度拥有独立的生成算法。

现代武装系统： 将引入具备弹道计算、AoE 爆炸破坏和后坐力的枪械与迫击炮。

复杂生活模拟与 AI： 包含多步骤的烹饪状态机，以及基于体素网格自定义寻路的宠物/怪物 AI 系统。

【开发架构约束（严格遵守）】

极致的内存与性能控制： 绝对禁止使用基于 GameObject 实例化的方块。必须采用底层数据数组配合 Procedural Mesh Generation（程序化网格生成）。对于区块生成等高密度计算，必须采用多线程（Task/ThreadPool）或 Compute Shader 异步处理，绝不能阻塞主线程。

数据与逻辑解耦： 大量使用 ScriptableObject 作为数据容器。

面向对象设计： 核心系统（如物品、方块、实体）必须基于清晰的 Interface 和抽象基类，确保后续添加新物品或新方块时符合开闭原则（OCP）。

如果你理解了项目的整体愿景和架构约束，请回复‘架构已确认’，我们将在下一步开始设计底层的体素数据结构。”

阶段一：深挖底层引擎 (原版方块与物理交互复现)
复现原版体验的核心在于流畅的无限世界和精准的方块判定。不要依赖 Unity 自带的物理引擎去处理成千上万的方块碰撞，那样开销极大。我们需要引导 AI 建立高效的数据结构。

Prompt 1.1 - 极致压缩的体素数据结构与区块管理：

“现在开始阶段一。请帮我设计体素引擎的最底层数据结构。

设计一个 Block 结构体。为了优化内存缓存命中率，不要在里面存引用类型。使用位操作（Bit-packing）或 byte/ushort 来存储 Block ID 即可。

设计 Chunk 类，尺寸为 16x256x16。使用一维数组展平三维坐标以提高连续内存访问速度（提供 xyz 到 index 的转换公式）。

设计 World 类作为单例或核心管理器，用字典（Dictionary）根据 Vector2Int（或 Chunk 坐标）来管理和检索当前加载的 Chunk。”

Prompt 1.2 - 异步地形生成与贪婪网格优化（Greedy Meshing）：

“基于刚才的数据结构，实现地形的生成与渲染。

编写一个基于多线程任务（Task）的生成逻辑，使用多重柏林噪声（Perlin/Simplex Noise）叠加，生成包含石头、泥土、草地的基础地形。

核心要求： 编写区块网格生成算法。除了基础的隐藏相邻面（剔除内部面）算法外，请尝试实现贪婪网格算法（Greedy Meshing），将同一平面上相同材质的相邻方块合并为一个大 Quad，以极大地降低 Vertex 和 Triangle 的数量。

提供将生成的 Mesh 数据分配给 Chunk GameObject 上的 MeshFilter 和 MeshCollider 的主线程回调逻辑。”

Prompt 1.3 - 自定义 AABB 物理与玩家交互：

“现在实现玩家控制与方块交互。

编写一个第一人称控制器。注意： 尽量不完全依赖 Unity 的 Rigidbody。请手写基于 AABB (Axis-Aligned Bounding Box) 与体素网格数据的碰撞检测逻辑，这样玩家在贴着墙壁移动或跳跃时才会有原版 MC 那种干脆、不滑动的物理反馈。

实现精准的射线检测（Raymarching）。从摄像机发射射线，遍历射线穿过的体素，返回击中方块的三维整数坐标以及击中的具体面（Normal Vector）。

实现左右键破坏与放置方块，并触发对应 Chunk（以及相邻 Chunk，如果在边界的话）的网格异步重建。”

Prompt 1.4(补充)
现在游戏的整体地图和前后移动的逻辑基本是完整的，但是存在几个问题：
1、地图方块的材质加载有问题，没有形成类似表面草方块/中层泥土/下层石头的地形渲染，整体地图更像是远看白色近看浅灰/透视的奇怪渲染
2、人物移动有一些问题，目前依靠的是自动的跳跃/下方块逻辑，背对着按s键走上方块（长按）会被判定成斜向后上方“飞起”，而不是正常移动。
3、人物和方块的交互：目前先实现对准方块长按左键1s会作为方块的破坏逻辑，后续会根据需要添加放置逻辑。
4、帮我限制一下地图范围，大概1000*1000*20格子就可以

阶段二：系统运转 (原版背包与合成复现)
在这个阶段，我们需要构建一套类似于原版支持 NBT (Named Binary Tag) 扩展潜力的系统，目前只需要实现9个格子物品数据的动态存储与加载（多余的放到背包中，最多45个物品），要求能够左键挖掘让方块变成掉落物，识别掉落物体拾取到背包中，合并同类项，右键可以放置物品。

Prompt 2.1 - 支持动态数据的物品基类架构：

“开始阶段二：物品与背包。请设计一套高扩展性的物品数据架构。
创建基于 ScriptableObject 的 ItemData 基类（包含 ID, Name, Icon, MaxStackSize）。。
设计 Inventory 类，维护一个 ItemStack 数组，提供 AddItem(ItemStack), RemoveItem(), HasItem() 等核心业务逻辑接口，逻辑必须与 UI 完全分离。”

Prompt 2.2 - 拖拽交互式背包 UI 系统：

“基于 Inventory 类，实现一套 Unity UGUI 界面。
创建固定数量的 Slot（槽位）UI 预制体。
实现复杂的鼠标交互逻辑：
左键点击：拿起/放下整个 ItemStack。
右键点击：拿起一半/放置时只放置一个物品（类似于原版的拆分逻辑）。
拖拽过程中的图标跟随鼠标渲染。
使用事件总线（或 C# Action 委托）让 UI 订阅数据层的变化，只有在数据改变时才刷新 UI。”

Prompt 2.3 - 形状无关的 3x3 合成匹配算法：

“最后，实现经典的 3x3 工作台合成台逻辑。
r键打开背包之后，可以显示5*9=45个物品的完整背包，在左上方是一个人物的贴图，右上方是一个合成区，合成区是一个3x3的网格，每个槽位可以放一个物品。定义 Recipe (ScriptableObject) 类，包含一个 3x3 的输入矩阵（可用一维数组表示空缺与特定物品 ID）和一个输出的 ItemStack。
核心算法难点： 编写合成匹配检测器。玩家在 3x3 UI 槽位中放入物品时，算法不仅要对比 ID，还需要支持空间平移匹配（例如：配方在最左侧一列，玩家把材料摆在最右侧一列，依然算作有效合成）。
实现点击输出槽位拿取产物时，自动扣除 3x3 矩阵中对应原材料数量的逻辑。“
